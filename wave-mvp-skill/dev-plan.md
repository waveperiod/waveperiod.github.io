# Wave 开发计划 (Dev Plan)

## 总体策略

### 核心原则
1. **先跑起来，再完善**
2. **每天交付可见成果**
3. **灵活调整，不死磕计划**

### 里程碑
- Day 1结束：能看到基础页面
- Day 3结束：核心功能能用
- Day 5结束：上线可访问

---

## 弹性调整建议

### 如果进度超前
- ✅ 增加P1功能
- ✅ 打磨细节
- ✅ 多做测试

### 如果进度落后
- ✅ 砍P1功能
- ✅ 简化交互
- ✅ 减少视觉打磨

### 紧急情况
Day 3结束时评估：
- 如果核心功能还没跑通 → 全力做P0
- 如果P0都完成了 → 可以加P1

## 每日工作流

### 早上开始前
```
1. 看一眼features.md，知道今天做什么
2. 问自己：今天结束时要交付什么？
3. 激活对应角色开始工作
```

### 遇到问题时
```
1. 先尝试自己解决（搜索、文档）
2. 5分钟解决不了，问我
3. 问题格式：
   - 我在做什么
   - 遇到什么问题
   - 期望结果是什么
   - 已经尝试了什么
```

### 晚上结束时
```
1. 更新features.md的状态
2. 记录今天的关键决策
3. 列出明天要做的3件事
```

## 风险预警

### 可能遇到的坑
1. **Chart.js配置复杂** → 先用默认配置
2. **ICS格式不对** → 参考examples
3. **移动端样式乱** → 用Tailwind响应式class
4. **GitHub Pages不生效** → 检查仓库设置

### 备用方案
- Plan A不行 → 立即切Plan B
- 不要在一个问题上卡超过2小时
